# Design notes — детали реализации

Дополнение к [../requirements.md](../requirements.md). Здесь — продуманные детали по принятым решениям (БД, батчинг записи), их последствия и открытые вопросы.

---

## 1. БД: SQLite вместо Postgres (решено)

### Почему это работает для нас
- Корпоративный мессенджер = умеренная нагрузка (десятки–сотни одновременных пользователей), один узел. Для этого SQLite более чем достаточно.
- Ноль инфраструктуры: нет отдельного сервиса БД, бэкап = копия одного файла, проще self-host.
- `sqlx` поддерживает SQLite (compile-time проверка запросов сохраняется).

### Что принимаем как trade-off
- **Один писатель.** SQLite сериализует записи. Конкурентные транзакции записи выстраиваются в очередь → это и есть причина батчить (см. ниже).
- **Один узел.** SQLite привязывает БД к одному процессу/машине. Горизонтальное масштабирование БД и HA — недоступны. Для нашего кейса ок; если позже понадобится — миграция на Postgres (поэтому держим SQL переносимым, избегаем чисто-SQLite-фич где дёшево).
- **WS fan-out тоже в одном процессе** (tokio broadcast-каналы на комнату). Это согласуется с одноузловой SQLite — не вводим Redis/брокер раньше времени.

### Обязательные PRAGMA при старте
```sql
PRAGMA journal_mode = WAL;        -- читатели не блокируют писателя
PRAGMA synchronous  = NORMAL;     -- с WAL: без corruption, теряется максимум последняя незакоммиченная транзакция
PRAGMA busy_timeout = 5000;       -- ждать, а не падать на SQLITE_BUSY
PRAGMA foreign_keys = ON;
PRAGMA wal_autocheckpoint = 1000; -- контроль роста WAL
```

### Пулы соединений
- **Один writer-коннект** (или пул=1) — единственный путь записи, владеет им выделенная задача-писатель.
- **Отдельный read-only пул** (несколько коннектов) — благодаря WAL читатели работают параллельно с писателем.

## 2. Батчинг записи сообщений (решено)

Цель: одна транзакция = один fsync на **пачку**, а не на каждое сообщение. На SQLite это даёт кратный рост throughput на пиках.

### Триггер флаша
Флашим пачку, когда наступает **первое из**:
- прошло **~50 мс** с момента первого сообщения в буфере, **или**
- накопилось **10–20 сообщений** (порог в конфиг, старт с 16).

### Схема (один писатель + канал)
```
WS-хендлеры ──mpsc(bounded)──> Writer task
                                  │  буфер Vec<Msg>
                                  │  tokio::select! { recv | sleep(50ms) }
                                  └─> BEGIN; INSERT ...×N; COMMIT  (одна транзакция)
                                  └─> оповестить ожидающих (ack)
```
- `tokio::select!` между `rx.recv()` и таймером; таймер взводится при добавлении **первого** сообщения в пустой буфер.
- При достижении порога по количеству — флашим немедленно, не дожидаясь таймера.
- **Bounded** канал = естественный backpressure: если писатель не успевает, отправители ждут (а не растёт память без предела).
- Один `INSERT` с несколькими VALUES (или подготовленный stmt в цикле) внутри одной транзакции.

### ID и порядок (важно для realtime)
- **ID генерируем в приложении — ULID** (сортируемый по времени, не палит счётчик, дружелюбен к SQLite). Это позволяет **разослать сообщение в комнату сразу**, не дожидаясь коммита в БД.
- `client_msg_id` (UUID от клиента) храним для **идемпотентности** — дедуп при реконнекте/ретрае, чтобы не задвоить сообщение.
- Сортировка истории — по ULID (он же монотонный по времени).

### Семантика доставки (решить явно)
Два варианта подтверждения отправителю:
- **Оптимистично (рекомендую для чата):** присвоили ULID+ts → сразу broadcast в комнату → персист батчем. UI отзывчивый. Минус: при падении до коммита возможна потеря последних ~50 мс сообщений (рассылка ушла, в БД нет).
- **Подтверждённо:** ack клиенту только после коммита батча. Надёжнее, но добавляет до ~50 мс латентности к «галочке доставлено».
- Компромисс: broadcast сразу, но «доставлено/сохранено» (вторая галочка) — после коммита. Окно потери при краше ≈ интервал батча; задокументировано и принято.

### Что НЕ батчим
- **Presence (онлайн-статус)** — держим в памяти (map socket→user), не персистим в БД на каждый чих.
- **Сессии/критичные операции** (выпуск/отзыв токена клиента, регистрация) — обычная синхронная транзакция, не через батч-канал.

## 3. Схема БД (черновик)

| Таблица | Назначение | Заметки |
|---|---|---|
| `users` | сотрудники | логин, password_hash (argon2), created_at |
| `sessions` | сессии сотрудников | token_hash, user_id, expires_at |
| `clients` | анонимные клиенты | display info, created_at |
| `client_tokens` | ссылки-токены клиентов | token_hash (не plaintext!), client_id, revoked_at, rotated_from |
| `rooms` | комнаты | type: `common` \| `client`, client_id (для per-client) |
| `room_members` | членство/доступ | room_id, user_id |
| `messages` | сообщения | **id ULID**, room_id, author (user/client), body, client_msg_id, created_at — **сюда пишем батчами** |
| `attachments` | метаданные вложений | message_id, filename, content_type, size, storage_key (сам файл — на диске/MinIO, не в SQLite) |
| `call_sessions` | звонки | room_id, started_at, ended_at |
| `call_recordings` | записи звонков | call_session_id, storage_path, format (opus/ogg), duration |

Индексы: `messages(room_id, id)` для пагинации истории; `messages(client_msg_id)` unique для идемпотентности; `client_tokens(token_hash)` unique; `sessions(token_hash)` unique.

**Вложения и записи звонков — НЕ в SQLite.** В БД только метаданные, бинарь — на диске или в MinIO (большие BLOB в SQLite раздувают файл и WAL).

## 4. Прочие продуманные детали (из requirements)

- **Токены клиентов:** хранить только хэш (как пароли); сравнение по хэшу. Ротация = новый токен + `revoked_at` старого, но `client_id` сохраняется → клиент возвращается в свой же диалог.
- **Реконнект WS:** клиент при реконнекте присылает `last_seen_id` (ULID) → сервер досылает пропущенное из БД. ULID-курсор делает это дёшево.
- **Лимиты вложений:** max size + whitelist content-type проверяем до записи на диск; превью картинок генерим асинхронно (не в горячем пути).
- **Rate limiting:** по IP и по сессии/токену; для анонимного входа — отдельно жёстче (защита от перебора токенов — но токен длинный и случайный, основная защита — энтропия + rate limit на эндпоинт входа).
- **WebRTC:** медиа через сервер, публичный IP → STUN/ICE нужен, TURN-релей не нужен. Выбор движка (webrtc-rs vs LiveKit) ещё открыт — влияет на то, как делать серверную запись (из RTP-трека вручную vs LiveKit Egress).

## 6. Стикеры: модель-ссылка + dotLottie (решено)

Стикеры шлём **ссылкой, а не байтами**: `messages.sticker` хранит короткий id
(напр. `heart`), обе стороны рендерят **бандл-ассет** из `/assets/stickers/`. 0 байт
на диске за отправку (в отличие от вложений, которые копируют blob каждый раз),
мгновенно, дедуп «бесплатно». Превью для тостов/пушей — «Стикер».

Рендер: `Sticker.svelte` **форматно-агностичен** — по расширению выбирает движок:
`.json/.lottie` → **dotLottie** (ThorVG WASM, canvas), `.webm` → нативный `<video>`,
`.webp/.gif` → `<img>`. WASM **self-host** (`/assets/dotlottie-player-<ver>.wasm`, без
CDN — RU-safe), `freezeOnOffscreen` для грида. WebM-видео-стикеры (VP9+alpha) не
требуют библиотеки, но нет альфы на iOS/WKWebView; Lottie — основной. Стартовый набор —
Noto Animated Emoji (Apache-2.0).

## 7. Сохранёнки: модель-копия (решено)

«Сохранёнки» (VK-style личная коллекция картинок/видео) — **независимая копия**:
при сохранении сервер копирует blob под новый id и пишет свою строку (`saved_items`),
а не ссылается на исходное вложение. Поэтому сохранёнка **переживает удаление**
оригинального сообщения — не нужен подсчёт ссылок. Цена — дублирование на диске.
`delete_saved` чистит свой blob (`Storage::remove`); дедуп минимальный — по
`source_id` (тот же attachment не копируется дважды). Отправка = копия обратно в
обычное вложение комнаты → общий путь `send`. Просмотр — общий лайтбокс (свайп).
GC осиротевших blob'ов исходников — TODO (см. [ideas.md](ideas.md)).

## 8. Android-пуши (FCM), офлайн-only (решено)

Единственный транспорт до спящего Android-приложения — FCM (HTTP v1, JWT
сервис-аккаунта → OAuth). Санкционированное исключение из «no external infra».
Шлём только тем из аудитории комнаты, кто **offline** (`presence.is_online == false`),
не автору. Ключ — `/data/fcm-sa.json` (без него сервер работает как обычно, без
пушей). Токены в `push_tokens`; нативная регистрация из Tauri-Android (patch.sh,
SharedPreferences), отписка при смене сервера. **Ограничение:** presence — на
аккаунт, не на устройство → открытая вкладка на десктопе глушит пуш на телефон
(fix «per-device» — в ideas.md). Реакции шлют отдельный тихий нудж.

## 9. Мобильная навигация на History API (решено)

Мобайл (≤640px, сотрудник) — телеграм-модель: полноэкранный список чатов + экран
диалога поверх. Навигация на **History API** (`pushState` при открытии чата →
`popstate` возвращает в список). Жест «назад» на Android ходит по истории WebView,
поэтому: лендинг переходит на хост через `location.replace` (не оставляет entry →
не выкидывает на форму логина), а на корне `MainActivity` делает `moveTaskToBack`
(свернуть, не разлогин). Десктоп — прежняя шторка.

## 10. Кеш и сжатие статики из бинарника (решено)

Ассеты отдаёт сам Rust-бинарник (`rust-embed`), поэтому кеш/сжатие — тоже в нём,
без зависимости от прокси. `serve_asset`: **ETag** (sha256) + `304`, path-based
`Cache-Control` (`immutable` только для версионно-фингерпринченных — dotLottie-wasm;
`no-cache` для html/бандла — ревалидация дешёвая через 304). Сжатие — `tower-http
CompressionLayer` (br/gzip/zstd), Caddy не нужен для `encode`. Windows-память хоста
продублирована в файл от Rust (`save_host`/`load_host`), т.к. WebView2 роняет
localStorage лендинга на рестарте.
