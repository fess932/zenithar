## Треды на сообщения (как в Mattermost)

**Суть:** у любого сообщения можно открыть тред — отдельную ветку ответов,
которая не засоряет основную ленту. В таймлайне у корневого сообщения видно
бейдж «N ответов» + аватары участников; клик открывает панель треда (сбоку на
десктопе, на весь экран на мобиле), где видны корень и все ответы по порядку.

**Почему это следующий логичный шаг:** фундамент уже есть — `messages.reply_to`
(см. ответы-цитаты, [REQUIREMENTS.md §3](../REQUIREMENTS.md)). Тред — это, по сути,
группировка сообщений по «корню ветки» поверх той же ссылки.

**Что нужно сделать (прикидка):**

- **Модель.** Ввести «корень треда». Варианты:
  - (a) переиспользовать `reply_to` и считать корнем самый верхний предок
    (рекурсивный CTE) — без новых колонок, но дороже на чтение;
  - (b) денормализовать `thread_root` (id корня) на сообщении — дешёвое чтение
    ветки одним `WHERE thread_root = ?`, но надо проставлять при отправке.
    **Скорее (b)** — наш паттерн (ср. денормализация `author_name`).
- **Счётчики.** `reply_count` + `last_reply_at` на корне (или вычислять). Нужны
  для бейджа и сортировки тредов по свежести.
- **Что показывать в основной ленте.** Решить: ответы в треде по умолчанию
  *не* попадают в главную ленту (mattermost-режим), либо есть тумблер «также
  отправить в канал». Это влияет на запрос истории (`recent_messages` фильтрует
  `thread_root IS NULL` для главной ленты).
- **WS-протокол.** Новый фрейм для подписки на тред (`thread-open`/`thread-history`)
  и доставки ответов открытого треда; либо расширить существующий fan-out полем
  `thread_root`, чтобы клиент сам решал, куда класть сообщение.
- **Фронт.** Панель треда (overlay/боковая колонка), бейдж «N ответов» на корне,
  переиспользовать `Composer` для ответа в тред. Mobile-first: тред на весь экран.
- **Звонки/вложения** в треде — наследуют существующую логику комнаты.

**Открытые вопросы:**

- Уведомления «вы участник треда» (требует presence/нотификаций — это Фаза 7).
- Глубина: плоский тред (как mattermost — все ответы к корню на одном уровне,
  без вложенности) сильно проще, чем дерево. **Начать с плоского.**
- Поиск/пагинация длинных тредов (пересекается с пагинацией истории).

**Объём:** средний. Бэкенд — одна колонка + пара запросов + фрейм; фронт — новая
панель + бейдж. Без вложенного дерева и нотификаций укладывается в одну фазу.

## Совсем анонимный «прайс-чат» + ИИ-резюме заказа

**Суть:** отдельная публичная ссылка-«визитка», по которой **каждый заход создаёт
новую комнату** (а не одну стабильную, как у обычного клиента). Зашёл случайный
человек спросить «сколько стоит X» — получил свежий анонимный диалог, без
регистрации и без узнаваемой идентичности между визитами. После разговора (текст
и/или запись звонка) комната прогоняется через ИИ, который вычленяет **короткое
описание заявки**, и оно сохраняется в базе как метаданные комнаты — чтобы
менеджеры видели список «лидов» с понятным саммари вместо сырых логов.

**Чем отличается от текущей модели:** сейчас клиент = один принципал = одна
постоянная комната (см. [decision: rooms model], `ensure_client_room`). Здесь
наоборот — **эфемерный гость**: новый принципал + новая комната на каждый заход
по «публичной» ссылке. Пересекается с REST API Фазы 6 (`POST /api/v1/clients`
уже умеет создавать клиента+комнату+первое сообщение) — публичная ссылка это,
по сути, тот же вызов, но инициированный самим гостем через UI, а не интеграцией.

**Что нужно сделать (прикидка):**

- **Ссылка-«фабрика».** Новый тип входа: токен не привязан к принципалу, а при
  открытии создаёт нового `client` + комнату и сразу логинит. Нужен отдельный
  роут (напр. `/g/:token`) и флаг у такой ссылки, чтобы отличать от обычной
  стабильной (`/i/:token`). Защита от спама — rate-limit по IP (уже есть бакеты).
- **ИИ-резюме.** После закрытия диалога/звонка собрать транскрипт: текст из
  `messages` + (опц.) расшифровка Ogg-записи звонка (Whisper-подобный STT) →
  выжимка LLM в 1–2 строки. Вынести за трейт `Summarizer` (как `Storage`), чтобы
  провайдер (локальная модель / API) дропался конфигом; работа из РФ — учитывать
  доступность провайдера.
- **Хранение саммари.** Колонка `summary` (+ `summary_at`) на `rooms` или
  отдельная таблица `leads`. Показать в админке списком: время, саммари, ссылка
  на комнату/запись.
- **Когда триггерить.** По завершении звонка (у нас уже есть хук финализации
  записи, см. Фаза 5) и/или по кнопке «сформировать саммари» у менеджера, и/или
  по таймауту неактивности гостевой комнаты.

**Открытые вопросы:**

- Приватность/удержание: эфемерные гостевые комнаты копятся — нужна ретенция
  (TTL/архивация) и политика хранения записей и саммари.
- Стоимость STT+LLM на каждый звонок — батчить/ставить лимиты; возможно, саммари
  только по явному действию менеджера.
- Антиспам публичной ссылки (капча/он-IP лимит), иначе фабрика комнат — вектор
  abuse.

**Объём:** крупный — пересекает auth (ссылка-фабрика), запись (STT), внешний ИИ
(новый трейт + провайдер) и админ-UI. Естественно ложится как отдельная фаза
поверх REST API и записи звонков.

## Наблюдаемость: метрики, логи и статы звонков в админке

**Суть:** встроенный дашборд в админке — **только встроенными средствами, без
внешних сервисов** (никакого Prometheus/Grafana/Loki/Redis-сервера). Три панели:
**нагрузка сервера** (CPU/RAM/uptime), **качество звонков** в реальном времени
(потери пакетов, jitter, RTT, битрейт по каждой ноге) и **последние логи** (хвост,
фильтр по уровню/модулю). Всё живёт внутри одного бинаря.

**Принцип хранения:** in-memory кольцевые буферы для «живого» вида; для истории,
переживающей рестарт, — **встроенная Rust-native БД** (эмбеддед, в стиле Redis,
напр. `sled`), либо уже имеющийся SQLite. НЕ внешний стор. (См. принцип
`feedback-self-contained-no-external-infra`.)

**Чем делать (Rust, прикидка):**
- **Метрики:** фасад `metrics` (counters/gauges/histograms по коду: сообщения/с,
  активные WS, звонки, ошибки) — но **рендерим сами** (снапшот в память → JSON в
  админку), без `metrics-exporter-prometheus`/внешнего скрейпа.
- **WebRTC-статы:** модуль `rtc::statistics` (`RTCStatsReport`) — опрашивать
  per-PeerConnection в драйвере (`calls/peer.rs`/`mod.rs`), агрегировать по звонку
  (packets_lost, jitter, rtt, bitrate). Снапшот в `Call`, периодически (~1/с).
- **Логи в админке:** свой `tracing-subscriber` Layer с кольцевым буфером
  (последние N в памяти) + `GET /api/admin/logs` (только admin). Persist — в
  embedded БД, не в Loki.
- **Нагрузка хоста:** `sysinfo` (CPU/RAM/процессы) или чтение `/proc`.
- **Фронт:** вкладка в админке (рядом с `Principals.svelte`), мини-графики на
  Canvas/SVG **без CDN** (наш принцип — без внешних ассетов).

**Открытые вопросы:**
- in-memory кольцо (теряется при рестарте) vs embedded `sled`/SQLite (история
  переживает рестарт, но растёт — нужна ретенция/обрезка).
- Приватность логов (id/ip) — только под `admin`, не наружу.
- Перф: статы звонков не чаще ~1/с, чтобы не грузить драйверы.

**Объём:** средний. Метрики + sysinfo + лог-буфер + админ-эндпоинт + вкладка.
WebRTC-статы — самое ценное, завязано на `rtc::statistics`. Делать после того, как
звонки заработают.

### Развилка хранилища: GreptimeDB как опциональный бэкенд

Рассматривали [GreptimeDB](https://github.com/greptimeteam/greptimedb) — Rust-нативная
TSDB (SQL/PromQL/InfluxDB-line/OTLP). Мощно, но это **отдельный серверный процесс**,
не эмбеддед-библиотека → нарушает принцип self-contained
([[feedback-self-contained-no-external-infra]]) и тяжеловат для нашего объёма.

**Решение (уточнено):** GreptimeDB **встраиваем в наш `docker-compose.yml`** как
сервис — оператор всё равно делает один `docker compose up`, ставить ничего
вручную не надо (санкционированное исключение к self-contained, см.
[[feedback-self-contained-no-external-infra]] — планка «один compose», не «один
бинарь»). Запись держим **за трейтом** (как `Storage`), чтобы:
- минимальный деплой мог обойтись встроенным (in-memory + `sled`/SQLite) без
  greptime-сервиса;
- основной деплой использовал bundled GreptimeDB для истории/масштаба.

Сложности, которые надо учесть: запись по gRPC/OTLP — **fire-and-forget** с
таймаутами (сбой/недоступность метрик-стора не должен ронять app); схема
time-series + ретенция; дашборд рисуем **сами** в админке (запрос к её SQL/PromQL),
Grafana НЕ тащим; в compose — healthcheck/зависимость старта, том для данных,
ресурсные лимиты; доступность образа из РФ (зеркало/pin версии).

## Нативные приложения на Tauri (iOS / Android / Windows / macOS / Linux)

**Суть:** упаковать существующий фронт в нативные приложения через **Tauri v2**
(стабильные мобильные таргеты с конца 2024). Tauri рендерит **наш текущий Svelte
как есть** в системном WebView (WKWebView / WebView2 / Android System WebView /
WebKitGTK), а нативное — это плагины на Rust + вкраплениями Swift/Kotlin. Один код
на iOS, Android, Windows, macOS и Linux, плюс десктоп как самый зрелый таргет Tauri.

**Почему именно Tauri для этого репо (а не RN/Flutter/натив):**

- Приложение — **тонкий клиент**: вся логика на сервере, клиент = UI + WS-плумбинг
  (`chat.ts`) + WebRTC (`call.ts`). Идеальная форма для упаковки.
- **Фронт переиспользуется на 100%** — ноль переписывания UI (~1900 строк Svelte).
  WebRTC живёт в вебвью → `call.ts` работает без изменений.
- **Бэкенд уже на Rust → ядро Tauri на Rust → шарим крейт** (типы WS-фреймов,
  `models`, валидация) между `backend/` и приложением. Этого не дают ни RN, ни
  Flutter, ни натив.
- **Десктоп (mac/win/linux) бесплатно** тем же кодом, без форков.
- **Flutter исключён** ([[feedback-no-flutter-dart]]); React Native — единственная
  реальная альтернатива, но требует переписать UI и форкнуть call-слой (web/native/
  desktop), зато даёт готовый фоновый CallKit-звонок «из коробки».
- **Главный размен:** Tauri дешевле и чище во всём, КРОМЕ одного — **звонок при
  убитом приложении на мобиле**. Tauri загоняет WebRTC в вебвью (где этих фич нет),
  поэтому фоновую звонилку (CallKit/PushKit, Telecom/FCM) пишем **сами как плагины**,
  готового `tauri-plugin-callkit` нет. У RN это «поставил callkeep». **Реальный
  вопрос выбора: насколько критичен звонок, когда приложение свёрнуто/убито.**

**Раскладка директорий (НЕ мешать с текущим кодом):** Tauri-проект — **в отдельной
директории** (напр. `app/`), рядом с `backend/` и `frontend/`. `app/` указывает
`frontendDist`/`devUrl` на существующий `frontend/` (его и рендерит), а нативные
плагины и Rust-ядро живут только внутри `app/`. Шаренные с `backend/` типы выносим
в отдельный крейт (напр. `shared/` или workspace-member), на который зависят оба.

**ГЛАВНЫЙ ПРИНЦИП: веб первичен, Tauri его не калечит.** Веб всегда работает на
100% сам по себе — он **не знает и не зависит от Tauri** ни на сборке, ни в рантайме.
Tauri — опциональный *потребитель*, который подключается к стабильному шву. Как это
гарантируется:

- **Зависимость строго односторонняя.** `app/` (Tauri) зависит от `frontend/` и
  рендерит **тот же `frontend/dist`, что уходит в браузер**. `frontend/` про
  существование `app/` ничего не знает. Tauri структурно **не может сломать веб**,
  т.к. не правит его — только оборачивает и в рантайме доинъекчивает возможности.
- **Один шов-адаптер** (`frontend/src/lib/platform.ts`). Маленький интерфейс для
  того, что различается: `onIncomingCall(cb)`, `reportCallAccepted/Declined()`,
  `setAudioRoute(speaker)`, `registerPushToken()`. **Дефолт — веб-реализация**
  (текущее поведение: ринг по WS, `setSinkId`/no-op). Весь остальной код
  (`call.ts`, `Call.svelte`) общается **только с интерфейсом**, никогда с Tauri.
- **Рантайм-детект, не build-флаги.** На старте проверяем `window.__TAURI_INTERNALS__`;
  если есть — **динамически** (`await import('@tauri-apps/api/core')`) подгружаем
  нативный адаптер и подменяем им дефолтный. Веб-бандл **не импортирует
  `@tauri-apps/*` статически** → нет жёсткой зависимости, отдельный чанк грузится
  только внутри Tauri. `@tauri-apps/api` держим в `optional/devDependencies`.
- **Грейсфул-деградация в контракте.** У каждой нативной возможности есть веб-фолбэк
  = сегодняшнее поведение (входящий → WS `call-ringing`; роутинг → no-op; пуш →
  no-op). Нет адаптера → приложение ведёт себя ровно как нынешний веб.
- **Ядро не трогаем — нативка садится сверху.** `call.ts` уже правильной формы:
  стор `incoming`, `acceptCall()`/`declineCall()`. Tauri-мост просто **тоже**
  кормит `incoming.set(...)` и зовёт `acceptCall()` — поверх, не переписывая.
- **CI доказывает самодостаточность веба.** Пайплайн собирает и гоняет фронт **без
  установленного Tauri** и проверяет, что приложение стартует при `isTauri=false`.
  Это машинно охраняет правило «веб всегда 100% работает».
- **Порядок работ из принципа:** сначала фича делается и живёт **в вебе** (через
  шов с no-op/WS-фолбэком). Потом, *если нужно*, в `app/` добавляется нативный
  адаптер, реализующий тот же интерфейс. Правки в `frontend/` при этом — только
  расширение шва (добавить метод в интерфейс + веб-фолбэк), что само по себе не
  ломает веб.

**Что нужно сделать (прикидка):**

- **Шов-адаптер.** Завести `frontend/src/lib/platform.ts` с интерфейсом и
  веб-реализацией по умолчанию **до** любой работы по Tauri — это и есть точка,
  через которую всё дальше неинвазивно.
- **Каркас.** `app/` с Tauri v2, `tauri ios init` / `tauri android init`; devUrl →
  на dev-сервер `frontend/`. Desktop-таргеты идут почти даром.
- **Auth: cookie → решить режим загрузки.** Два варианта:
  - грузить приложение как **remote-URL** (`https://сервер`) → вебвью = вкладка
    браузера, **cookie работают как сейчас**, `session.ts` не трогаем (минус —
    нужна сеть, нет офлайн-шелла);
  - **бандлить** фронт локально (`tauri://localhost`) → запросы к API cross-origin
    → нужен **Bearer-токен** (выдача по link-deep-link) + CORS. Правка бэкенда.
  - Скорее **токен** — он же нужен для пушей и единообразен с мобилой.
- **Звонилка — два уровня амбиций:**
  - **Уровень 1 (звонит, пока приложение живо):** WS `call-ringing` уже работает
    → in-app UI + локальное уведомление + звук + фокус окна. На десктопе держим
    приложение в трее/меню-баре с живым WS (`tauri-plugin-autostart` + tray). Это
    **полностью закрывает Windows/macOS** и mobile-while-open. Быстро, без риска.
  - **Уровень 2 (звонит при убитом приложении на мобиле):** push-to-wake + родной
    call-UI. Основной R&D, делать последним.
- **Единый мост к JS (контракт одинаков на всех ОС):** натив эмитит в вебвью
  события `incoming-call` / `call-accepted` / `call-declined`, фронт дёргает
  существующие `acceptCall()` / `declineCall()` из `call.ts`. **Медиа-путь не
  трогаем — нативка только будит и показывает экран звонка.**
- **iOS (плагин, Swift):** `PKPushRegistry` (VoIP-токен → бэкенд); на VoIP-push
  **обязательно** немедленный `CXProvider.reportNewIncomingCall` (иначе iOS убьёт
  процесс) → родной CallKit-экран даже при убитом приложении; `CXAnswerCallAction`
  → emit `call-accepted`. `Info.plist`: `UIBackgroundModes=voip,audio`,
  `NSMicrophoneUsageDescription`.
- **Android (плагин, Kotlin):** FCM high-priority data → `setFullScreenIntent` +
  `CATEGORY_CALL` (экран звонка на локскрине), либо `TelecomManager`/ConnectionService.
  Разрешения: `POST_NOTIFICATIONS` (13+), `USE_FULL_SCREEN_INTENT` (14+),
  `FOREGROUND_SERVICE_MICROPHONE`; foreground service на время разговора.
- **Windows:** трей + autostart, на `call-ringing` — WinRT toast
  `scenario="incomingCall"` (звонит, стойкий toast, кнопки Принять/Отклонить),
  вероятно через крейт `windows` напрямую; активация → фокус окна + emit.
- **macOS:** меню-бар + autostart, `UNUserNotificationCenter` с действиями
  Принять/Отклонить + зацикленный ринг (`AVAudioPlayer` `numberOfLoops=-1`); accept
  → `NSApp.activate` + emit. (CallKit на macOS есть, но ограничен — берём
  уведомление.)
- **Микрофон в вебвью:** iOS — usage-string в Info.plist; Android — пробросить
  разрешение в вебвью через `onPermissionRequest`.
- **Аудио-роутинг (динамик/ухо):** WKWebView не умеет `setSinkId` (наш `call.ts`
  это и отмечает) → нужен маленький нативный плагин на AVAudioSession/AudioManager.
- **Бэкенд (общее для всех платформ):** хранить push-токены устройств (APNs VoIP /
  FCM); на `call-start` слать пуш спящим callee в дополнение к WS `call-ringing`.

**Что переиспользуется без изменений:** все `*.svelte`, Tailwind, `chat.ts`,
`call.ts`, `emoji/i18n/lightbox`. Меняем/добавляем: режим auth (возможно токен),
`localStorage` → нативный KV там, где нужен офлайн, и тонкий `incoming-call`-мост.

**Списки и видео (потолок вебвью — учитывать на будущее):**

- **Сложные/длинные списки** (лента сообщений) — **нативных компонентов в Tauri
  нет** (весь UI = вебвью). Решается **виртуализацией в вебе** (TanStack Virtual /
  `content-visibility`), одной реализацией на все ОС; для чата этого хватает с
  запасом. `Chat.svelte` дозреет до оконного рендеринга при длинной истории.
- **Видеозвонки** (если появятся) — рисуются **инлайн `<video>` в основном вебвью**,
  отдельное окно НЕ нужно (и не ускоряет — `RTCPeerConnection`/`MediaStream` нельзя
  шарить между вебвью-контекстами). «Открепляемый» звонок — через **Document
  Picture-in-Picture API** (тот же контекст), не через второе окно. Видео — в
  основном **серверная** работа (форвард видео-RTP + simulcast в sans-IO SFU),
  фронт меняется скромно.
- **Настоящая нативная стена** (толкает к RN/нативу только для конкретного экрана):
  виртуальные фоны/блюр нативного качества, очень большие видео-сетки, кастомные
  GL-эффекты. Базовый видеозвонок и небольшие сетки вебвью тянет.

**Открытые вопросы:**

- Режим загрузки (remote-URL vs bundled) — определяет auth и офлайн-поведение.
- Фоновые звонки на мобиле — самый дорогой и рискованный кусок (пишем нативные
  плагины сами); возможно, на первом этапе ограничиться уровнем 1.
- Зрелость WebRTC в системных вебвью на мобиле (особенно Android) — проверять
  на реальных устройствах: проброс разрешений, кодеки, стабильность.
- Доступность APNs/FCM и образов/тулчейна из РФ; подпись и публикация в сторы.

**Объём:** крупный, но поэтапный. **Уровень 1 (foreground-звонки + десктоп в трее)
— средний и без риска**, даёт рабочее приложение на всех 5 ОС быстро. **Уровень 2
(фоновые CallKit/PushKit + FCM/full-screen)** — основной R&D, делать последним,
после того как заработают звонки на сервере и базовая упаковка. Естественный
порядок: каркас `app/` → токен-auth + push-токены на бэкенде → уровень 1 везде →
iOS PushKit→CallKit → Android FCM→full-screen.

## Пуш-уведомления на уровне устройства (не аккаунта)

**Проблема.** Сейчас offline-push (`send::push_offline`) отсекает получателей по
`presence.is_online(principal_id)`, а презенс считает живые WS-соединения **на
аккаунт (principal)**, а не на устройство (см. докстринг `presence.rs`: «several
connections (tabs/devices); only 0↔1 transitions emit»). Поэтому если один и тот
же аккаунт открыт на компе (вкладка с живым WS) и на телефоне (приложение
закрыто) — аккаунт считается online, и **на телефон пуш не уходит**. Презенс
знает «аккаунт где-то online», но не знает, что конкретно это устройство offline.

(Замечено на тесте: уведомления не приходили, пока на десктопе была открыта
вкладка того же аккаунта. Закрыл вкладку — приходят. Т.е. ведёт себя «по
дизайну», но дизайн неполный для мультидевайса.)

**Вариант A — презенс/пуш по токену устройства (правильный).** Клиент при
WS-хендшейке сообщает свой push-токен; сервер держит набор «online-токенов», пока
сокет жив. `push_offline` шлёт на токены аудитории (кроме автора), **у которых нет
живого сокета** — независимо от других устройств аккаунта. Бонус: уходит и фактор
«presence залип» на негладком обрыве, т.к. смотрим на конкретный сокет. Стоимость:
прокинуть токен в WS-коннект + хранить online-токены (по аналогии с
`PresenceRegistry`, но ключ — токен/устройство).

**Вариант B — решает клиент (дешевле).** Слать `data`-only пуш на **все** токены
аудитории (кроме автора), а показывать уведомление решает само устройство: рисуем
в `onMessageReceived`, только если приложение **не на переднем плане**. Презенс
аккаунта вообще не при чём, каждое устройство ведёт себя по своему состоянию.
Минусы: больше логики на клиенте, теряется авто-рисование системой (`notification`
payload), возможны дубли (пуш + внутриаппный тост) если не аккуратно с состоянием.

Предпочтение — **A**. См. [[decision-android-push-fcm]].
