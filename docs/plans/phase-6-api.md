# Фаза 6 — REST API для интеграций

Внешний REST API, чтобы интеграции (CRM, боты) читали/писали в чат и заводили
клиентов. Аутентификация — постоянные **Bearer-токены**, выпускаемые из админки.

## Модель

- Новый вид принципала **`bot`** (интеграция) — наряду с `user`/`client`. Бот:
  - полноценный автор сообщений (в ленте видно его имя),
  - «staff»: доступ ко всем комнатам (как сотрудник), см. `auth::is_staff`,
  - инертен вне API: не подключается по WS, не висит в presence,
  - не показывается в списке ссылок (`/api/principals`) — у него своя секция.
- **`api_tokens`** — отдельная таблица (не link-токены). Храним только SHA-256;
  плейнтекст показывается один раз. Формат токена — `zk_<base64url(32 байта)>`.
  Поля: `name`, `created_at`, `last_used_at` (best-effort), `revoked_at`.

## Аутентификация

Экстрактор `ApiAuth` (`auth.rs`): читает `Authorization: Bearer zk_…`, проверяет
per-token rate-limit (бакет по хэшу токена), резолвит активный токен → принципал,
best-effort обновляет `last_used_at` (в фоне, не блокирует запрос).

- 401 — нет/невалидный/отозванный токен.
- 429 — превышен лимит (60 burst, +5/с на токен).

CSRF/Origin-проверка к API не применяется (это не cookie-поток; non-browser
клиенты без `Origin` всё равно проходили бы). Управляющие ручки интеграций
(`/api/integrations…`) — наоборот, cookie + `Admin` + origin-check.

## Эндпоинты `/api/v1` (Bearer)

| Метод | Путь | Назначение |
|------|------|-----------|
| GET  | `/api/v1/me` | кто я (id/name/kind) — проверка ключа |
| GET  | `/api/v1/rooms` | все комнаты (common + клиентские, с `client_id`) |
| GET  | `/api/v1/rooms/{id}/messages?limit&before` | история, oldest-first, пагинация назад по `before` (id сообщения) |
| POST | `/api/v1/rooms/{id}/messages` | отправить в комнату `{body, reply_to?, attachment_ids?}` |
| POST | `/api/v1/clients/{client_id}/messages` | то же, но адресуясь по `client_id` клиента (room резолвится сам) |
| POST | `/api/v1/clients` | создать клиента: `{name?, order?}` → `{client_id, room_id, url}` |
| POST | `/api/v1/uploads` | multipart `room_id`+`file` → метаданные вложения (его `id` затем в `attachment_ids`) |

`limit` клампится 1..=100 (по умолчанию 50). Отправка переиспользует тот же путь,
что и WebSocket (`send::deliver`): broadcast подключённым + (для клиентских
сообщений) пинг сотрудникам + батч-запись. То есть сообщения из API видны в
реальном времени и сохраняются идентично.

### Создание клиента + первое сообщение

`POST /api/v1/clients` с `{name, order}`:
1. создаёт принципала `client` (имя = `name`, случайное если пусто),
2. заводит ему комнату,
3. выпускает link-токен → `url` = `/i/<token>` (относительный; хост подставляет
   вызывающий),
4. если `order` непустой — постит его **от имени клиента** первым сообщением
   (читается как входящая заявка и пингует сотрудников).

Это и есть интеграционный сценарий: CRM создаёт клиента, получает шарящуюся
ссылку, а краткое описание заказа появляется первым сообщением в чате.

## Управление (админка, cookie + Admin)

| Метод | Путь | |
|------|------|--|
| GET  | `/api/integrations` | список интеграций (name, created_at, last_used_at, active) |
| POST | `/api/integrations` | создать `{name}` → `{id, name, token}` (токен один раз) |
| POST | `/api/integrations/{id}/rotate` | новый токен, старый отзывается |
| POST | `/api/integrations/{id}/revoke` | отозвать токен(ы) |

Во фронте — секция «Интеграции» в админ-панели (`Principals.svelte`): создать,
скопировать токен один раз, ротация, отзыв, индикатор «использован».

## Вне v1 (задел)

- Гранулярные scopes (сейчас токен = полный доступ интеграции).
- Выдача записей звонков наружу.
- «Ссылка-фабрика» для эфемерных гостей + ИИ-резюме заказа — см.
  [docs/ideas.md](../ideas.md) (естественно ложится поверх этого API и записи).

## Замечания по схеме

`api_tokens` + расширение CHECK на `principals.kind` требуют пересоздания БД
(один dev-файл миграции): `make db-reset`.
