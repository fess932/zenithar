# Фаза 1 — Авторизация (план, ред. 2)

Цель: ввести идентичность. **Паролей нет.** И сотрудники, и клиенты входят
по постоянной ссылке-токену (link-as-auth) — механизм один и тот же. Отличие —
роль принципала: `user` (сотрудник) или `client` (аноним). Ссылки создаёт
**админ из UI** (галочка «пользователь / анонимный»). Новому принципалу даётся
случайное имя; после входа сотрудник сам меняет имя. После фазы WebSocket и
сообщения перестают быть анонимными — `author` берётся из подтверждённой
идентичности.

Связано: [../../requirements.md](../../requirements.md) §1, [../design-notes.md](../design-notes.md).

---

## 1. Подтверждённые решения

- **D1. Сессия сотрудника** — cookie + серверная сессия (opaque-токен в
  httpOnly-cookie, строка сессии в SQLite, хранится хэш).
- **D2. Создание учёток** — **без логина/пароля**. Админ создаёт принципалов из
  UI; механизм входа — ссылка-токен (как у анонимов), роль задаётся флагом
  `user|client`. Случайное имя при создании → самостоятельное переименование
  после входа.
- **D3. Идентичность клиента/пользователя** — токен в ссылке → обмен на httpOnly
  cookie при первом заходе; ссылка остаётся долговечной (повторный резолв).
- **D4. Ротация токена** — один активный токен на принципала; reissue отзывает
  прежний, `principal_id` сохраняется (тот же диалог/идентичность); старые строки
  остаются для аудита (`rotated_from`).

Бутстрап первого админа — **авто при первом старте**: если в БД нет ни одного
принципала, сервер создаёт админа (`kind=user`, `is_admin=1`), выпускает токен и
печатает ссылку входа `/i/<token>` в лог. Дальше админ создаёт остальных из UI.

Открытый мелкий вопрос: переименование для клиентов — **нет** (они анонимны,
имя опционально задаёт админ); самопереименование — только для `user`.

---

## 2. Модель / схема БД

Правим **единый** `backend/migrations/0001_init.sql` (БД пересоздаём `make db-reset`).

Унифицируем сотрудников и клиентов в одну таблицу принципалов:

- `principals`
  - `id` (ULID)
  - `kind` `TEXT CHECK (kind IN ('user','client'))`
  - `display_name` (случайное при создании; `user` может менять)
  - `is_admin` `INTEGER` (0/1; только для `user`)
  - `created_at`
- `tokens` — долговечные ссылки-токены (для обоих kind)
  - `id`, `token_hash` UNIQUE, `principal_id` FK, `created_at`,
    `revoked_at` NULL, `rotated_from` NULL
- `sessions` — cookie-сессии (обмен токена на cookie)
  - `token_hash` UNIQUE, `principal_id` FK, `created_at`, `expires_at`, `last_seen`
- `messages` — `author` заменяем на `author_id` (FK principals) +
  денормализованный `author_name` для отображения. `author_kind` берём из принципала.

Индексы: `tokens(token_hash)`, `tokens(principal_id)`, `sessions(token_hash)`.
«Один активный токен» обеспечиваем логикой (rotate ставит `revoked_at` прежнему).

---

## 3. Зависимости (Rust)

- `rand` — CSPRNG для токенов сессий и ссылок (32 байта → base64url ≈ 43 симв.).
- `sha2` — хэш токенов/сессий при хранении и поиске.
- `axum-extra` (feature `cookie`) или `tower-cookies` — cookie.
- `time` — сроки сессии/cookie.
- Генератор имён — маленький встроенный список (прилагательное+существительное),
  без новой зависимости.
- (argon2 **не нужен** — паролей нет.)

---

## 4. Бэкенд: эндпоинты

Вход / идентичность:
- `GET /i/:token` — резолв ссылки-токена (по SHA-256). Валиден и не отозван →
  создаёт сессию, ставит cookie `zsid`, отдаёт SPA. Иначе 404/410.
- `POST /api/auth/logout` — удаляет сессию, чистит cookie.
- `GET  /api/me` — `{kind, id, display_name, is_admin}` или `null`.
- `POST /api/me/name` `{display_name}` — самопереименование (только `user`).

Админ (только `is_admin`):
- `POST /api/principals` `{kind:'user'|'client', display?}` → создаёт принципала
  (если имя не задано — случайное) + выпускает токен, отдаёт **ссылку**
  `{principal_id, url:"/i/<token>"}` (токен показывается один раз).
- `POST /api/principals/:id/rotate` → отзывает прежний, выпускает новый, ссылка.
- `POST /api/principals/:id/revoke` → `revoked_at`.
- `GET  /api/principals` → список (без токенов: kind, имя, активна ли ссылка, даты).

WebSocket:
- `/ws` гейтится: извлекаем `Identity` из cookie `zsid` → принципал.
  Нет идентичности → отказ при апгрейде (401). `author_id/author_name` сообщений
  берём из идентичности; поле от клиента игнорируем. (Роутинг по комнатам — Фаза 2.)

---

## 5. Бэкенд: структура

- `auth/` модуль: `tokens.rs` (gen/hash/compare), `session.rs` (create/lookup/revoke),
  `principal.rs` (create/list/rotate/revoke/rename), `names.rs` (генератор имён).
- Экстракторы axum (`FromRequestParts`): `Identity` (из `zsid`),
  `Admin` (Identity + is_admin), для WS — тот же `Identity`.
- Конфиг (env): `ZENITHAR_SECURE_COOKIES` (bool), срок сессии (по умолч. 30 дней).
- Авто-бутстрап админа при старте, если БД пустая (печать ссылки в лог).

---

## 6. Фронтенд (Svelte)

- При загрузке — `GET /api/me`:
  - есть сессия (`user`) → чат + имя в шапке (клик → редактирование),
    кнопка «Выйти»; если `is_admin` — доступна панель ссылок.
  - есть сессия (`client`) → чат как клиент (аноним).
  - нет сессии → лендинг «Откройте свою ссылку-приглашение» (формы входа нет —
    вход только по ссылке).
- Маршрут `/i/:token`: бэкенд поставил cookie и отдал SPA → фронт видит из
  `/api/me`, кто он. (Ссылку можно затем «почистить» из адресной строки.)
- Первый вход с случайным именем → ненавязчивый баннер «Задайте своё имя».
- Панель админа `Principals.svelte`: создать ссылку (**галочка пользователь/аноним**),
  список принципалов, «Перевыпустить»/«Отозвать», показ свежей ссылки + копирование.
- Локализация (RU/EN) и Tailwind — как в Фазе 0; новые строки в `i18n.ts`.

## 7. Безопасность

- Токены ссылок и сессий — только SHA-256-хэш в БД; сравнение по хэшу.
- Энтропия токена ≥ 256 бит — основная защита входа по ссылке (паролей нет).
- Cookie `HttpOnly`, `SameSite=Lax`, `Secure` за TLS (флаг), `Path=/`.
- Мутирующие `POST` — проверка `Origin` (базовая защита от CSRF; `SameSite=Lax`
  закрывает основное).
- Базовый троттлинг резолва `/i/:token` по IP (полный rate limiting — Фаза 7).
- Отозванный/просроченный токен → отказ; logout инвалидирует сессию.

---

## 8. Тесты

- Unit: gen+hash токена; resolve отозванного → отказ; генератор имён.
- Integration (axum): `/i/:token` создаёт сессию и cookie; `/api/me` по cookie;
  logout инвалидирует; create/rotate/revoke принципала (admin-гейт); rename только
  для `user`; WS-апгрейд без cookie → 401.
- e2e (bun): админ-ссылка → вход как admin user → создаёт ссылку клиента → клиент
  по `/i/<token>` входит → оба видят WS со своей идентичностью.

## 9. Шаги реализации

1. Зависимости (rand, sha2, axum-extra/cookies, time) + конфиг cookie.
2. Схема в `0001_init.sql` (principals/tokens/sessions, messages author_*) + db-хелперы.
3. Утилиты: генерация/хэш токенов, генератор имён.
4. Авто-бутстрап админа при старте (если БД пуста) + печать ссылки в лог.
5. `/i/:token` (резолв→сессия→cookie), `/api/auth/logout`, `/api/me`, `/api/me/name`.
6. Экстракторы `Identity`/`Admin`; гейт `/ws`; author из идентичности.
7. `/api/principals*` (create/list/rotate/revoke).
8. Фронт: гейт по `/api/me`, лендинг, имя+переименование, кнопка выхода,
   панель `Principals.svelte` с галочкой user/anon.
9. Тесты (unit + integration + e2e).
10. Docs: README (бутстрап админа, создание ссылок), отметки в requirements/design-notes.
