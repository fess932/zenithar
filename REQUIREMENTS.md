# Корпоративный мессенджер со звонками — требования

Self-hosted сервис общения. Назначение: внутренний инструмент компании + пет-проект на GitHub.
Стек: **Rust** (бэкенд) + **TypeScript** (фронтенд).

---

## 1. Пользователи и доступ

- [ ] **Сотрудники** — зарегистрированные пользователи (логин + сессия).
- [ ] **Клиенты** — анонимные пользователи, без регистрации.
- [ ] **Вход клиента по ссылке-токену:**
  - [ ] Постоянная ссылка, токен в URL = удостоверение (link-as-auth).
  - [ ] Одна стабильная ссылка на клиента → всегда возвращает в его же диалог.
  - [ ] Токен длинный и случайный.
  - [ ] Возможность отозвать / перевыпустить токен конкретного клиента.

## 2. Комнаты

- [ ] **Общая комната** — командный чат сотрудников.
- [ ] **Комната per client** — клиент попадает по своей ссылке, любой сотрудник может ответить.
- [ ] Логика роутинга: входящий клиентский диалог виден сотрудникам, любой подхватывает.

## 3. Чат

- [ ] Реалтайм-сообщения (WebSocket).
- [ ] Вложения: картинки и файлы (загрузка, хранение, отдача, превью).
- [ ] Лимиты на размер и проверка content-type.

## 4. Звонки

- [ ] **Только голос** (без видео). Качество — достаточное для речи.
- [ ] **WebRTC** (решено: TCP-релей даёт скачки задержки на потерях — не подходит).
- [ ] Медиа идёт **через сервер**: client → server → client (сервер в середине).
- [ ] TURN не нужен для самого релея — у сервера публичный IP, он всегда достижим (нужны базовые STUN/ICE).
- [ ] AEC / шумоподавление / AGC — через констрейнты `getUserMedia` на захвате.

## 5. Запись звонков

- [ ] Серверная запись (не на клиенте).
- [ ] Аудио-only, сохранение **на сервере**.
- [ ] Формат: Opus / Ogg (или согласно выбранному медиа-движку).

## 6. API

- [ ] REST API для интеграций.

## 7. Self-hosted

- [ ] Разворачивается на своей инфраструктуре (Docker / docker-compose).
- [ ] Документация по развёртыванию.

---

## Технические решения

- **Бэкенд:** Rust — axum/actix + tokio; Postgres (sqlx); WebSocket для чата + WebRTC-сигналинга.
- **Фронтенд:** TypeScript — React или Svelte; браузерный WebRTC; чат-UI; загрузка файлов.
- **Хранилище:** диск или S3-совместимое (например self-hosted MinIO) — для вложений и записей.
- **Медиа (WebRTC, сервер в середине)** — выбрать один из путей:
  - **Вариант A — `webrtc-rs`:** свой SFU-lite на чистом Rust. Максимум контроля, лучший пет-проект, больше работы (DTLS/SRTP, RTP, запись из трека).
  - **Вариант B — LiveKit (self-hosted):** SFU + Egress (серверная запись аудио в S3) из коробки. Минимум кода, но +инфраструктура (LiveKit + Egress + Redis). Есть Rust server SDK.

## Решения, которые надо принять до старта

- [ ] WebRTC: `webrtc-rs` (свой) vs LiveKit (готовый).
- [ ] Фронт-фреймворк: React vs Svelte.
- [ ] Хранилище: локальный диск vs S3/MinIO.
- [ ] Политика ротации/отзыва клиентских токенов.

---

## Дорожная карта (по фазам)

- [ ] **Фаза 0 — Скелет:** структура репо, Rust-сервер с WS, схема БД, базовый TS-фронт.
- [ ] **Фаза 1 — Авторизация:** сессии сотрудников + токен-ссылки клиентов.
- [ ] **Фаза 2 — Чат и комнаты:** общая + per-client, реалтайм по WS.
- [ ] **Фаза 3 — Вложения:** загрузка / хранение / отдача / превью.
- [ ] **Фаза 4 — WebRTC:** сигналинг + медиа через сервер (голос).
- [ ] **Фаза 5 — Запись:** серверная запись аудио, сохранение на сервере.
- [ ] **Фаза 6 — REST API.**
- [ ] **Фаза 7 — Прод-обвязка:** переподключения, presence (онлайн-статус), rate limiting, безопасность анон-входа, деплой.

## Нефункциональные требования

- [ ] Переподключение WebSocket / звонка при обрыве.
- [ ] Presence — статус «онлайн».
- [ ] Rate limiting.
- [ ] Безопасность анонимного входа по токену.
- [ ] Docker-развёртывание + инструкция self-host.
