# Корпоративный мессенджер со звонками — требования

Self-hosted сервис общения. Назначение: внутренний инструмент компании + пет-проект на GitHub.
Стек: **Rust** (бэкенд) + **TypeScript** (фронтенд).

---

## 1. Пользователи и доступ

> **Реализовано в Фазе 1, беспарольно.** Все входят по ссылке-токену (см.
> [docs/plans/phase-1-auth.md](docs/plans/phase-1-auth.md)); роль `user`/`client`
> задаётся при создании. Учётки заводит админ из UI; первый админ — авто-бутстрап.

- [x] **Сотрудники** (`user`) — создаются админом из UI; вход по ссылке-токену
      (как у клиентов), сессия в httpOnly-cookie. Имя случайное, меняется самим.
- [x] **Клиенты** (`client`) — анонимные, без регистрации; вход по своей ссылке.
- [x] **Вход по ссылке-токену** (общий механизм для обоих):
  - [x] Постоянная ссылка, токен в URL = удостоверение (link-as-auth) → обмен на cookie.
  - [x] Одна стабильная ссылка на принципала → всегда тот же диалог/идентичность.
  - [x] Токен длинный и случайный (≥256 бит), в БД только SHA-256-хэш.
  - [x] Отзыв / перевыпуск токена конкретного принципала (из UI).

## 2. Комнаты

> **Реализовано в Фазе 2.** Модель и протокол → [docs/plans/phase-2-rooms.md](docs/plans/phase-2-rooms.md).

- [x] **Общая комната** — внутренний командный чат, только для сотрудников.
- [x] **Комната per client** — клиент попадает по своей ссылке, любой сотрудник может ответить.
- [x] Логика роутинга: клиентский диалог виден всем сотрудникам, любой подхватывает (без назначений).

## 3. Чат

- [x] Реалтайм-сообщения (WebSocket).
- [x] Вложения: картинки и файлы (загрузка, хранение, отдача, превью) — Фаза 3,
      хранилище за `Storage`-трейтом (диск; S3 позже), серверные тумбнейлы картинок.
- [x] Голосовые сообщения — запись через `MediaRecorder`, как аудио-вложение, `<audio>` в ленте.
- [x] Эмодзи — встроенный офлайн-пикер (нативные эмодзи, без CDN; работает из РФ).
- [x] Лимиты на размер (25 МБ) и проверка content-type (сниффинг картинок).
- [ ] Стикеры — набор картинок-ассетов (открытый набор локально), пока не сделано.

## 4. Звонки

- [ ] **Только голос** (без видео). Качество — достаточное для речи.
- [ ] **WebRTC** (решено: TCP-релей даёт скачки задержки на потерях — не подходит).
- [ ] Медиа идёт **через сервер**: client → server → client (сервер в середине).
- [ ] TURN не нужен для самого релея — у сервера публичный IP, он всегда достижим (нужны базовые STUN/ICE).
- [ ] AEC / шумоподавление / AGC — через констрейнты `getUserMedia` на захвате.

## 5. Запись звонков

- [ ] Серверная запись (не на клиенте).
- [ ] Аудио-only, сохранение **на сервере**.
- [ ] Формат: Opus / Ogg (или согласно выбранному медиа-движку).

## 6. API

- [ ] REST API для интеграций.

## 7. Self-hosted

- [ ] Разворачивается на своей инфраструктуре (Docker / docker-compose).
- [ ] Документация по развёртыванию.

## Технические решения

- **Бэкенд:** Rust — axum/actix + tokio; **SQLite (sqlx)** — решено, см. [docs/design-notes.md](docs/design-notes.md); WebSocket для чата + WebRTC-сигналинга.
- **Фронтенд:** TypeScript — React или Svelte; браузерный WebRTC; чат-UI; загрузка файлов.
- **Хранилище:** диск или S3-совместимое (например self-hosted MinIO) — для вложений и записей.
- **Медиа (WebRTC, сервер в середине)** — выбрать один из путей:
  - **Вариант A — `webrtc-rs`:** свой SFU-lite на чистом Rust. Максимум контроля, лучший пет-проект, больше работы (DTLS/SRTP, RTP, запись из трека).
  - **Вариант B — LiveKit (self-hosted):** SFU + Egress (серверная запись аудио в S3) из коробки. Минимум кода, но +инфраструктура (LiveKit + Egress + Redis). Есть Rust server SDK.

## Решения, которые надо принять до старта

- [x] **БД: SQLite (sqlx)** — решено. Запись сообщений батчами (~50мс или 10–20 сообщений). Детали и последствия → [docs/design-notes.md](docs/design-notes.md).
- [ ] WebRTC: `webrtc-rs` (свой) vs LiveKit (готовый).
- [x] **Фронт-фреймворк: Svelte** (Svelte 5 + TypeScript, сборка через bun + bun-plugin-svelte).
- [x] **Хранилище: локальный диск за `Storage`-трейтом** — решено (Фаза 3); S3/MinIO дропается позже без переписывания вызовов.
- [x] Политика ротации/отзыва токенов: один активный токен на принципала; reissue
      отзывает прежний (тот же principal_id → тот же диалог), история через `rotated_from`.

## Дорожная карта (по фазам)

- [x] **Фаза 0 — Скелет:** структура репо, Rust-сервер с WS, схема БД, базовый TS-фронт.
- [x] **Фаза 1 — Авторизация:** беспарольный вход по ссылке-токену → cookie-сессия,
      унифицированные принципалы (user/client), управление ссылками из UI, авто-бутстрап админа.
- [x] **Фаза 2 — Чат и комнаты:** общая + per-client, реалтайм по WS.
- [x] **Фаза 3 — Вложения:** загрузка / хранение / отдача / превью; голосовые; эмодзи.
- [ ] **Фаза 4 — WebRTC:** сигналинг + медиа через сервер (голос).
- [ ] **Фаза 5 — Запись:** серверная запись аудио, сохранение на сервере.
- [ ] **Фаза 6 — REST API.**
- [ ] **Фаза 7 — Прод-обвязка:** переподключения, presence (онлайн-статус), rate limiting, безопасность анон-входа, деплой.

## Нефункциональные требования

- [ ] Переподключение WebSocket / звонка при обрыве.
- [ ] Presence — статус «онлайн».
- [ ] Rate limiting.
- [ ] Безопасность анонимного входа по токену.
- [ ] Docker-развёртывание + инструкция self-host.
